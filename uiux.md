# 會議室預借系統 UX/UI 規劃

---

## 1. 使用者與情境摘要

### 主要使用者類型

| 使用者類型 | 心理模型 | 到達時的情緒狀態 | 主要目標 |
|-----------|---------|----------------|---------|
| **一般員工（申請人）** | 習慣 Google Calendar / Outlook；預期「選好時間、選好房間、送出就好」 | 通常有時間壓力，希望快速完成，不想填複雜表單 | 確認到空房 → 送出申請 → 知道有沒有過 |
| **主管 / 核准人** | 每天收到多封通知，期望一鍵決策不需跳頁 | 忙碌、通知疲勞，希望減少點擊 | 快速審核申請、不遺漏待辦 |
| **行政管理員** | 管理多個場地與規則，需要全局視角 | 需要掌控全局，怕衝突與重複預借 | 維護房間、處理例外、產出使用報表 |

### 使用者核心任務（依重要性排序）

1. 快速找到「這個時段有哪間會議室是空的」
2. 提出預借申請並追蹤進度
3. 查看我目前的所有預借（已核准、待審、已拒絕）
4. 取消或修改既有預借
5. （管理員）審核申請、管理房間設定

### Anti-goals（使用者不想碰到的事）

- 填完才被告知時段已被預訂
- 不知道申請卡在哪個環節
- 看到過時的空閒狀態（資訊不即時）
- 審核頁面要反覆跳頁才能完成決策
- 手機操作時按鈕太小、行事曆難以使用

## 2. 使用者流程地圖

### 流程 A：員工申請預借會議室

```
[入口：首頁儀表板 / 導覽列「預借會議室」]
  │
  ▼
Step 1：選擇日期與時段
  → 使用者看到：月曆 + 時段選擇器（AM/PM 區塊或 30 分鐘拖拉）
  → 系統即時顯示：該時段可用的會議室清單（灰色 = 不可用）
  ✓ 快樂路徑：選好日期時段，可用房間即時更新
  ✗ 邊緣案例：所選時段全部滿房 → 顯示「此時段無空房，建議改為 [最近可用時段]」

Step 2：選擇會議室
  → 使用者看到：房間卡片（房間名稱、容納人數、設備標籤、縮圖）
  → 點擊卡片可展開詳細資訊（設備列表、位置樓層、當日時間表）
  ✓ 快樂路徑：選定房間，點「申請此會議室」進入表單
  ✗ 邊緣案例：使用者想先比較多間 → 支援「加入比較」或 Drawer 不跳頁預覽

Step 3：填寫申請資訊
  → 使用者看到：精簡表單（會議主題、參與人數、用途說明、是否需要設備）
  → 申請人資訊自動帶入（姓名、部門、分機）
  ✓ 快樂路徑：填寫後按「送出申請」
  ✗ 邊緣案例：送出時該時段剛被搶走 → 頁面頂部顯示 Banner 錯誤，保留已填資料

Step 4：確認送出
  → 使用者看到：申請摘要確認彈窗（房間、時段、主題）
  → 確認送出後系統建立申請單，發送通知給核准人
  ✓ 系統導向「我的預借」頁，新申請顯示「審核中」狀態
  ✗ 網路錯誤 → 提示重試，防止重複送出

[完成：進入「我的預借」查看申請狀態]
```

---

### 流程 B：主管審核申請

```
[入口：Email 通知連結 / 系統通知鈴鐺]
  │
  ▼
Step 1：查看待審清單
  → 顯示申請人、房間、時段、用途摘要
  → 每列提供「核准」「拒絕」按鈕，不需進詳情頁

Step 2：決策操作
  ✓ 直接核准 → 申請人收到核准通知，行事曆更新
  ↳ 拒絕 → 彈出輸入框填寫拒絕原因（必填，最少 10 字）
  ↳ 想看更多資訊 → 展開列或跳至詳情頁

[完成：待審清單數字歸零]
```

---

### 流程 C：員工查看我的預借

```
[入口：導覽列「我的預借」]
  │
  ▼
  → 分頁標籤：全部 / 審核中 / 已核准 / 已拒絕 / 已完成
  → 每筆顯示：房間、日期時段、申請狀態、操作按鈕
  ↳ 已核准 → 可「取消預借」（24 小時前）
  ↳ 審核中 → 可「撤回申請」
  ↳ 已拒絕 → 顯示拒絕原因，提供「重新申請」（帶入原資料）
```

## 4. 互動模式與元件規格

### 4.1 即時可用性顯示

- **模式名稱**：Optimistic real-time availability
- **觸發**：使用者選擇日期或拖動時段
- **行為**：選擇完成後 300ms 內，房間列表依可用狀態重新渲染。可用 = 全色卡片；不可用 = 灰底 + 「已預訂」標籤，仍可點開查詳情
- **理由**：讓使用者在選房前就能排除不可用選項，避免「選了才發現沒空」的挫折感

---

### 4.2 房間詳情：Drawer（側拉抽屜）而非跳頁

- **模式名稱**：Right-side Drawer preview
- **觸發**：點擊房間卡片
- **行為**：畫面右側滑出 Drawer，寬度 480px（桌機）/ 全螢幕（手機）。Drawer 內顯示房間照片、設備清單、當日時間軸（橫軸顯示哪些時段已被佔用）、「申請此會議室」按鈕
- **理由**：保持背景的房間列表可見，使用者看完詳情後可直接關閉 Drawer 選其他房間，不需按上一頁，減少導航成本
- **替代方案**：跳頁詳情頁 — 缺點是返回後列表需重新載入，使用者失去比較脈絡

---

### 4.3 申請表單：Progressive Disclosure（漸進揭露）

- **模式名稱**：Progressive form disclosure
- **觸發**：進入申請表單頁
- **行為**：
  - 第一屏只顯示必填欄位（會議主題、預計人數）
  - 「需要額外設備？」為可展開區塊，預設收合
  - 「備註說明」為選填，視覺層級較低（灰色 label）
- **理由**：降低表單視覺複雜度，讓使用者感覺「這很快就能填完」

---

### 4.4 審核操作：Inline Action（列表內操作）

- **模式名稱**：Inline approve / reject
- **觸發**：主管進入待審清單
- **行為**：
  - 每列顯示「核准」（綠色）與「拒絕」（紅色邊框）按鈕
  - 點「核准」→ 直接送出，列表移除該筆並顯示成功 Toast（3 秒）
  - 點「拒絕」→ 該列展開一個 inline 輸入框（非 Modal），填寫原因後確認送出
- **理由**：避免主管需要逐一進詳情頁操作，10 筆申請可在 30 秒內處理完。拒絕使用 inline 展開而非 Modal，避免遮蔽其他列資訊

---

### 4.5 衝突提示：頁頂 Banner（非中斷式 Alert）

- **模式名稱**：Non-blocking error banner
- **觸發**：送出申請時，後端偵測到時段剛被其他人搶走
- **行為**：表單頁頂部出現橘色 Banner「此時段的 [會議室名稱] 剛剛已被預訂，請重新選擇時段」，表單資料保留，使用者可直接修改時段重送
- **理由**：不清空表單、不強制 Modal，讓使用者付出最小成本修正錯誤

---

## 5. 摩擦點與 UX 風險

| 風險 | 嚴重度 | 說明 | 緩解設計 |
|------|-------|------|---------|
| **時段衝突（Race Condition）** | 高 | 兩人同時選同房間同時段，其中一人送出後另一人不知情 | 送出時即時驗證；失敗後顯示 Banner 保留資料（見 4.5） |
| **行事曆可用性資訊過時** | 高 | 使用者看到的空檔在送出時已被佔用 | 頁面停留超過 2 分鐘自動 refetch；房間時間表顯示最後更新時間 |
| **主管通知疲勞、漏審** | 高 | 通知太多導致忽略，申請卡住 | 逾期 24 小時未審核自動發送再次提醒；儀表板顯示「待審 N 件」紅色角標 |
| **使用者不知道申請進度** | 中 | 送出後沉默，不知道是否有人在審 | 送出後頁面即跳至「我的預借」，審核中項目有 Skeleton 動畫並顯示「等待 [審核人姓名] 審核」 |
| **手機端時段選擇困難** | 中 | 30 分鐘精度的拖拉時段在手機上難以操作 | 手機改用下拉選單（開始時間 + 結束時間）取代拖拉；桌機保留拖拉 |
| **取消預借沒有確認機制** | 中 | 誤觸取消無法還原 | 取消需二次確認 Modal，顯示「確定取消 2026/04/10 14:00–16:00 的 A101 預借？」 |
| **表單欄位說明不清楚** | 低 | 「用途說明」不知道要寫什麼 | 欄位下方顯示 placeholder 範例：「例：季度業績檢討，約 8 人，需要投影機」 |

## 6. 無障礙與包容性

### 鍵盤導航
- 日期選擇器支援方向鍵移動日期、Enter 確認、Esc 關閉
- 房間卡片可用 Tab 聚焦，Enter / Space 展開 Drawer
- Drawer 開啟時焦點自動移入，關閉後焦點回到觸發元素

### 螢幕閱讀器
- 狀態 Badge 加上 `aria-label`（不能只靠顏色傳達意義）
- 行事曆時間軸提供表格語義標記（`<table>` + `scope` 屬性），確保 VoiceOver / NVDA 可讀
- 操作成功 / 失敗的 Toast 訊息使用 `role="alert"` 確保播報

### 色彩與視覺
- 所有文字對比度 ≥ 4.5:1（WCAG AA）
- 狀態不單靠顏色區分，同時搭配文字標籤與圖示
- 支援瀏覽器字體放大至 150% 不破版

### 觸控與手機
- 所有可點擊元素觸控區域 ≥ 44×44px
- 列表操作按鈕在手機版改為底部 Action Sheet，避免誤觸
- 支援 iOS Safari / Android Chrome 主流版本

### 國際化
- 日期格式依系統語系顯示（zh-TW：YYYY/MM/DD）
- 時間使用 24 小時制
- 預留 i18n 架構，字串統一管理於語系檔

## 7. 成功指標

| 指標 | 目標值 | 量測方式 |
|------|-------|---------|
| **申請完成率** | ≥ 85%（進入選房頁到成功送出） | 前端 funnel 事件追蹤 |
| **申請流程平均耗時** | ≤ 2 分鐘（找到房間 → 送出） | 前端計時事件 |
| **主管平均審核時間** | ≤ 4 小時 | 申請建立時間 vs. 審核完成時間 |
| **使用者滿意度（CSAT）** | ≥ 4.0 / 5.0 | 完成申請後顯示 1 題評分（可略過） |

## 8. 開放問題與後續步驟

| 問題 | 優先度 | 建議行動 |
|------|-------|---------|
| 是否需要「即時預借」（當下就可進去，不需審核）還是「所有申請皆需審核」？ | 高 | 與業務確認審核政策後決定流程分支 |
| 多層審核（申請人 → 部門主管 → 行政）是否需要？ | 高 | 確認組織層級，設計對應的審核流程引擎 |
| 週期性預借（每週四 14:00 固定借 A101）是否在範圍內？ | 中 | 若需要，表單需加入重複規則設計 |
| 會議室使用中是否需要簽到功能（防止爽約佔位）？ | 中 | 若有簽到需求，需設計超時自動釋放機制 |
| 外部訪客預借（非系統帳號）是否需要支援？ | 低 | 確認後決定是否加入訪客申請流程 |
